# 再発防止：ダッシュボードに反映されない・保存が2回必要

このドキュメントは、以下のような事象を防ぐための**開発時のチェックリスト**です。

- 求職者を「日程調整中」などにしてもダッシュボードに反映されない
- 読み金額を求職者ページで入力しても、1回ではダッシュボードに出ない（2回入力すると出る）

---

## 利用者（担当者）が気をつけること

※ 2026年3月の修正で「2回入力しないと出ない」「ステータスが反映されない」は解消済みです。以下は万が一のときの確認用です。

| 気をつけること | 説明 |
|----------------|------|
| **保存したあとは「ダッシュボードを開き直す」** | 求職者ページでステータスやヨミ（読み金額）を保存したあと、ダッシュボードを見るときは、一度ダッシュボードの画面を開き直す（メニューから「ダッシュボード」をクリックする）と、最新が表示されます。別タブで開いていた場合は、そのタブをクリックして画面に戻ると自動で更新されます。 |
| **「反映されていない」ときは画面を更新** | 保存したのにダッシュボードに数字や名前が出ていないときは、ブラウザの更新（F5 または ⌘+R）を1回試してください。 |
| **2回入力する必要はない** | 読み金額は1回保存すればダッシュボードに反映される仕様です。以前「2回入力したら出た」事象は不具合として修正済みです。 |

---

## 事象と原因の対応関係

| 事象 | 原因のパターン | 防ぐためのルール |
|------|----------------|------------------|
| ステータスを変えてもダッシュボードに出ない | 「〇〇が0件なら表示しない」でスキップしている | **0件でも「あり」として表示する**か、仕様で明示する |
| 金額を1回保存してもダッシュボードに出ない | PATCH で「送っていない項目」まで null で上書きしている | **PATCH は「送った項目だけ」更新する**（ホワイトリスト） |
| 別画面で更新して戻ると古いまま | 再取得を「初回だけ」にしている | **フォーカス・タブ表示時にも再取得**する |
| ヨミ保存後に面接追加すると金額が0円になる | 面接追加で新プロジェクトが作られ、ヨミなしのプロジェクトをダッシュボードが表示する | **面接追加時は既存ヨミを引き継ぐ**＋**ヨミ保存時は全プロジェクトに反映**＋**ダッシュボードはヨミありプロジェクトを優先** |

---

## 開発時のチェックリスト（再発防止）

### 1. 一覧・ダッシュボードの「表示条件」を触るとき

- [ ] **「〇〇が0件なら return / スキップ」を入れていないか**
  - 例: 「プロジェクトが0件の求職者」→ ステータスだけでも出したいなら、0件でも行として出す（金額0・ヨミなしでよい）。
- [ ] **「代表1件」の選び方が他画面と揃っているか**
  - 求職者ページは「先頭1件」、ダッシュボードは「updated_at 最新1件」など**違うルール**にすると、編集した案件と表示される案件がずれる。同じルールにする。
- [ ] **同一求職者に複数プロジェクトがある場合、ヨミ情報を持つプロジェクトが優先されるか**
  - 面接追加で新プロジェクトが作られると、ヨミ情報のないプロジェクトが最新になりうる。ヨミ（probability + expected_amount）があるプロジェクトを優先して表示する。

### 2. 保存 API（PATCH）を触るとき・新規で作るとき

- [ ] **body をそのまま `update({ ...body })` や `update(body)` に渡していないか**
  - 渡すと「送っていないキー」が undefined や null で上書きされることがある。→ **更新してよいフィールドをホワイトリストで列挙し、`body` にそのキーがあるときだけ `updateRow` に積む**。
- [ ] **id や candidate_id など、変更してはいけない項目を更新していないか**
  - ホワイトリストに含めない。

**良い例（ホワイトリスト）：**

```ts
const ALLOWED_PATCH_FIELDS = ['expected_amount', 'probability', 'probability_month'] as const
const updateRow: Record<string, unknown> = { updated_at: new Date().toISOString() }
for (const key of ALLOWED_PATCH_FIELDS) {
  if (rawBody[key] !== undefined) updateRow[key] = rawBody[key]
}
await supabase.from('projects').update(updateRow).eq('id', id)
```

### 3. 「他画面で更新したあと戻ってくる画面」を触るとき

- [ ] **初回マウント以外で再取得しているか**
  - ダッシュボードなど「最新を表示したい画面」では、`visibilitychange`（タブ切替）や `window.focus`（ウィンドウに戻ったとき）でも `fetchData` を呼ぶと、今回のような「反映されない」を防ぎやすい。

## テストで確認するとよいこと（手動・E2E）

リリース前や大きな変更後に、以下をさっと確認すると再発に気づきやすいです。

1. **ステータス反映**
   - 求職者詳細でステータスを「面接日程調整中」に変更 → ダッシュボードの面接状況に**すぐ**その名前が出るか（案件が0件の求職者でも出るか）。
2. **読み金額反映**
   - 求職者詳細のヨミで「金額」だけ入力して保存 → ダッシュボードの面接状況・ヨミ数字に**1回の保存で**反映されるか。園名・法人名が消えていないか。
   - **ヨミ保存→面接追加の順で操作しても**、ダッシュボードに金額が0円にならないか。
3. **成約額反映**
   - 成約一覧で売上（税抜）を編集して保存 → ダッシュボードの売上数字に反映されるか。
