# コードレビューコマンド

## ロール

Hoiku CRM プロジェクト（Next.js 16 App Router / React 19 / TypeScript / Supabase / Tailwind CSS / shadcn-ui）のコードを多角的にレビューし、セキュリティ、設計、品質、パフォーマンス、規約の観点から包括的なフィードバックを提供する。

## 優先順位

1. **セキュリティ**: 脆弱性やセキュリティリスクの防止を最優先
2. **正確性**: コードが想定通りに動作し、エッジケースも適切に処理されているか
3. **設計原則**: 関心の分離、高凝集、疎結合が保たれているか
4. **コード品質**: 型安全性、可読性、保守性
5. **パフォーマンス**: 明らかな非効率な処理がないか
6. **規約遵守**: プロジェクトの規約と既存パターンに従っているか

## レビューフロー

### 1. 事前確認

- 対象ファイル・変更範囲の特定
- 影響範囲の確認（循環参照、共有ユーティリティの有無）
- プロジェクトのドキュメント（`.cursor/`、`README.md`、`docs/` など）との整合性確認
- 未コミット変更部分を重点的に確認

### 2. 多層的チェック

以下の順序で体系的にレビューを実施：

1. **セキュリティチェック** → 2. **設計原則チェック** → 3. **正確性・動作チェック** → 4. **コード品質チェック** → 5. **技術スタック特化チェック** → 6. **パフォーマンスチェック** → 7. **プロジェクト規約チェック**

### 3. フィードバック作成

- 優先度付け（Critical/High/Medium/Low）
- 具体的な修正案の提示（Before/Afterコード例を含む）
- 「なぜ」の説明を必ず含める
- 影響範囲とリスクを明記

### 4. 確認ポイント

レビュー完了時に以下を確認：
- TypeScript 型チェック（`tsc --noEmit`）
- ESLint チェック（`npm run lint`）
- ビルド確認（`npm run build`）
- E2E テスト（`npm run test:e2e`、変更が関係する場合）

## レビューチェックリスト

### 1. セキュリティチェック（最優先）

#### 1.1 データアクセス制御
- [ ] フロントエンドから Supabase に直接接続していないか（必ず `src/app/api/` ルート経由か）
- [ ] 全 API ルートで `src/lib/auth/server.ts` による認証チェックが実装されているか
- [ ] Supabase RLS（Row Level Security）が適切に機能しているか
- [ ] `src/lib/supabase/server.ts`（サーバー用）と `src/lib/supabase/client.ts`（クライアント用）が正しく使い分けられているか

#### 1.2 ビジネスロジックの配置
- [ ] 重要なビジネスロジック（ステータス遷移、集計、判定）がフロントエンドに露出していないか
- [ ] `src/app/api/` 以下のルートハンドラーでビジネスロジックが実装されているか

#### 1.3 機密情報の管理
- [ ] `NEXT_PUBLIC_` プレフィックスを付けていい情報かどうか確認済みか
- [ ] Supabase Service Role Key がクライアントサイドに露出していないか
- [ ] Google Sheets API 認証情報（`src/lib/google-sheets.ts`）がサーバー側のみで使用されているか
- [ ] ログ出力に個人情報（求職者氏名、連絡先など）が含まれていないか

#### 1.4 入力値の検証
- [ ] API ルートで zod スキーマによる入力バリデーションが実装されているか
- [ ] `react-hook-form` + `@hookform/resolvers/zod` によるフォームバリデーションが実装されているか
- [ ] Supabase クエリにユーザー入力を直接連結していないか（パラメータバインディング使用）

#### 1.5 認証・認可
- [ ] `src/middleware.ts` で保護すべきルートが適切に保護されているか
- [ ] 権限（admin/member など）によるアクセス制御が適切か

### 2. 設計原則チェック

#### 2.1 関心の分離
- [ ] Server Component / Client Component の境界が適切か（`'use client'` の使用箇所）
- [ ] データ取得は Server Component または API ルートで行われているか
- [ ] UIロジックとビジネスロジックが分離されているか
- [ ] カスタムフック（`src/hooks/`）でデータ取得ロジックが分離されているか

#### 2.2 高凝集・疎結合
- [ ] 関連する機能が同じ場所にまとまっているか
- [ ] コンポーネント間の依存関係が最小限になっているか
- [ ] 循環参照がないか

#### 2.3 重複排除
- [ ] 重複コードがないか（DRY原則）
- [ ] `src/lib/utils.ts` や既存ユーティリティが活用されているか
- [ ] `src/types/database.ts` の既存型定義が流用されているか

#### 2.4 ファイル構造
- [ ] `src/app/` 以下のディレクトリ構造が App Router の規約に従っているか
- [ ] `src/components/` の命名・配置がプロジェクトの規約に従っているか
- [ ] 同一関心のファイルが3件以上になった場合、専用ディレクトリが作成されているか

### 3. 正確性・動作チェック

#### 3.1 基本動作
- [ ] コードは想定通りに動作しているか
- [ ] エッジケース（空配列、null/undefined、ネットワークエラー）が適切に処理されているか
- [ ] Supabase クエリのエラーハンドリングが実装されているか
- [ ] ローディング・エラー状態のUIが考慮されているか

#### 3.2 型安全性
- [ ] `any` 型や型アサション（`as`）の不適切な使用がないか
- [ ] `src/types/database.ts` の型定義が適切に使用されているか
- [ ] Supabase のレスポンス型が適切に扱われているか
- [ ] zod スキーマと TypeScript 型の整合性が取れているか

### 4. コード品質チェック

#### 4.1 React / Next.js のベストプラクティス
- [ ] Hooks のルール（最上位での呼び出し、条件分岐なし）を遵守しているか
- [ ] `useEffect` の依存配列が適切か（過不足ないか）
- [ ] `key` プロパティが一意で安定した値を使用しているか
- [ ] Server Actions または API ルートを適切に選択しているか
- [ ] `next/image`、`next/link` などの Next.js 最適化コンポーネントが使用されているか

#### 4.2 コード整理
- [ ] 未使用コード・インポートが削除されているか
- [ ] 変数・関数に意味のある名前が付与されているか（日本語コメントとの整合性）
- [ ] マジックナンバーが定数化されているか
- [ ] `console.log` などのデバッグコードが残っていないか

#### 4.3 shadcn/ui・Radix UI の使用
- [ ] shadcn/ui コンポーネント（`src/components/ui/`）が適切に使用されているか
- [ ] Radix UI のアクセシビリティ属性が維持されているか

### 5. 技術スタック特化チェック

#### 5.1 Next.js 16 App Router
- [ ] `'use client'` ディレクティブが必要な箇所にのみ付与されているか
- [ ] `loading.tsx` / `error.tsx` が適切に配置されているか（必要な場合）
- [ ] Route Handler（`route.ts`）のレスポンス形式が統一されているか
- [ ] `next/headers` の cookies が適切に使用されているか

#### 5.2 Supabase
- [ ] `src/lib/supabase/queries-client.ts` や `queries-server.ts` の既存クエリが再利用されているか（`queries.ts` は非推奨）
- [ ] リアルタイムサブスクリプションのクリーンアップが実装されているか
- [ ] Supabase Storage の公開/非公開バケットの使い分けが適切か

#### 5.3 Google Sheets 連携（GAS / `src/lib/google-sheets.ts`）
- [ ] Google Sheets との同期ロジックが `src/lib/sync-candidates.ts` のパターンに従っているか
- [ ] GAS（`GAS_SYNC.gs`）と整合性が取れているか

#### 5.4 Tailwind CSS v4
- [ ] Tailwind のユーティリティクラスが適切に使用されているか
- [ ] `cn()` ユーティリティ（`src/lib/utils.ts`）で条件付きクラスが管理されているか
- [ ] カスタムスタイルが必要な場合は `globals.css` の `@theme` で管理されているか（Tailwind v4）

### 6. パフォーマンスチェック

#### 6.1 レンダリング最適化
- [ ] 不要な `'use client'` によるクライアント化が行われていないか
- [ ] 重い処理が Server Component で行われているか
- [ ] `useMemo` / `useCallback` が適切な箇所で使用されているか（過剰使用に注意）

#### 6.2 データ取得の最適化
- [ ] 不要なデータ取得・ウォーターフォールリクエストがないか
- [ ] Supabase クエリで必要なカラムのみ `select()` しているか
- [ ] N+1 問題が発生していないか

#### 6.3 バンドルサイズ
- [ ] 大きなライブラリが動的インポート（`dynamic()`）で遅延読み込みされているか
- [ ] 不要な依存が追加されていないか

### 7. プロジェクト規約チェック

#### 7.1 スタイルガイド
- [ ] ESLint（`eslint-config-next`）のルールに準拠しているか
- [ ] TypeScript の strict モードに準拠しているか（`tsconfig.json` 確認）
- [ ] ファイル名はケバブケース（`kebab-case.tsx`）になっているか

#### 7.2 既存パターン
- [ ] API ルートのエラーレスポンス形式が既存パターン（`{ error: string }`）に従っているか
- [ ] カスタムフック（`src/hooks/`）のパターンに従っているか
- [ ] `src/lib/supabase/queries-client-with-fallback.ts` のフォールバックパターンが必要か確認
- [ ] ステータス管理が `src/lib/status-mapping.ts` を通じて行われているか

#### 7.3 ドキュメント整合性
- [ ] `docs/` 以下のドキュメントと整合性が取れているか
- [ ] 重要な設計変更が `docs/` に反映されているか（必要に応じて）

## フィードバック構造

各フィードバック項目は以下の構造で記述：

```
**[優先度]** [カテゴリ] [タイトル]

**問題点:**
- 具体的な問題の説明（ファイルパス・行番号を含む）

**理由:**
- なぜこの問題が重要か

**修正案:**
- 具体的な修正方法
- Before/After コード例（必要に応じて）

**影響範囲:**
- この変更が影響する範囲・リスク
```

優先度の分類：
- **Critical**: セキュリティリスク、重大なバグ、動作しないコード
- **High**: 設計上の問題、パフォーマンス問題、型安全性の問題
- **Medium**: コード品質の問題、規約違反、改善の余地
- **Low**: スタイルの問題、細かい改善提案

## 禁止事項

- 仕様変更・UI変更・依存追加/削除・環境設定変更（明示指示がある場合を除く）
- 過度な抽象化や premature optimization の提案
- 未確認での命名・構造大規模変更の提案
- 挙動に影響しうる定数・条件分岐の改変（必要な場合は必ず理由とリスクを明記）

## 実装指針

- **シンプルを最優先**: オーバーエンジニアリング禁止
- **常に最小限で設計・実装**: 必要十分な機能のみを実装
- **ファイルの細分化を避ける**: 過剰な場合は統合を検討
- 未コミット変更部分（`git diff`）を重点的に確認
- リスクがあれば先に列挙
